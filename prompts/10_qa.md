# QA Design Prompt

You are an AI tutor designing a self-check Q&A section.

> ⏸ **On-demand only.** 신규 논문 기본 빌드 범위에서 제외됨. 사용자가 ⑥ 콘텐츠 작성을 명시 요청한 경우에만 실행한다 (CLAUDE.md "기본 빌드 범위" 정책).

## 역할

학습자가 **논문을 다 읽은 뒤** 자가 점검할 수 있는 Q&A 세트를 설계한다.

이건 평가 시험지가 아니다. **학습자가 혼자 읽었을 때 모호하게 남았을 만한 부분을 미리 짚어주는** 학습 보조 도구다.

---

## 표준 카테고리 구조 (정본)

3개 카테고리, 각각 2~3개 질문. 총 6~10 질문 권장.

| ID | 제목 (예시) | 다루는 것 |
|---|---|---|
| **A** | 기초 지식 / 지형도 | 논문 도메인의 큰 그림. 이 분야를 처음 보는 학습자가 길을 잃지 않게 하는 입문 |
| **B** | 메커니즘 | 논문 메서드의 작동 원리. "왜 이 단계가 필요한가" "왜 이 순서인가" 같은 인과 |
| **C** | 실험 해석 | 결과 표/그림을 어떻게 읽을 것인가. 수치의 함의, 함정, 비교의 정당성 |

이 3-카테고리는 모든 논문에 통용된다. 카테고리 ID는 `A`, `B`, `C` 고정이지만 제목은 논문에 맞게 조정 가능.

## JSON 스키마 (정본)

```json
{
  "categories": [
    {
      "id": "A",
      "title": "LVLM 추론의 지형도",
      "questions": [
        {
          "qid": "q1",
          "title": "한 프레임이 토큰이 되는 길",
          "user_question": "(선택) 학습자가 던질 법한 원래 질문 형태",
          "tldr": "1~2문장 한줄 요약 답변",
          "blocks": [
            { "type": "html",    "content": "..." },
            { "type": "table",   "headers": ["...", "..."], "rows": [["...", "..."]] },
            { "type": "math",    "tex": "...", "explanation": "..." },
            { "type": "callout", "variant": "key", "content": "..." }
          ]
        }
      ]
    },
    { "id": "B", "title": "...", "questions": [...] },
    { "id": "C", "title": "...", "questions": [...] }
  ]
}
```

### 필드 규약

- `qid`: `q1`, `q2`, ... 논문 전체 유일
- `title`: 한 줄 짧은 질문 (15자 이내 권장)
- `user_question` (선택): 학습자가 자연어로 던질 법한 형태 ("이건 왜 이렇게 되는 거예요?" 식). title보다 길어도 됨.
- `tldr`: TL;DR — 1~2문장으로 답을 압축. 본문을 다 읽지 않아도 이것만으로 핵심을 가져갈 수 있어야 함.
- `blocks`: 본문 블록 배열. 4가지 type만 허용 (`html`, `table`, `math`, `callout`).

### 블록 타입별 형식

#### `html`
```json
{ "type": "html", "content": "<p>...</p>" }
```
자유 마크업. 위젯 SVG도 여기 포함. 인라인 수식은 `$...$`.

#### `table`
```json
{
  "type": "table",
  "headers": ["조건", "FastV", "Ours"],
  "rows": [
    ["VideoMME", "63.4", "65.4"],
    ["...", "...", "..."]
  ]
}
```
간단한 비교 테이블. 5×5 이하 권장.

#### `math`
```json
{ "type": "math", "tex": "S_t = \\frac{X_{t-P}^\\top X_t}{\\|X_{t-P}\\|\\,\\|X_t\\|}", "explanation": "이 식은 ..." }
```
display 수식 + 한 줄 설명. JSON에서 `\\` 두 번 (`rules/math_rules.md` 참조).

#### `callout`
```json
{ "type": "callout", "variant": "key|warn|safe", "content": "..." }
```
강조 박스. `variant` 3종:
- `key`: 핵심 인사이트 (초록 계열)
- `warn`: 주의/함정 (빨강 계열)
- `safe`: 부드러운 부연 (회색/베이지 계열)

## 작성 원칙

### 1. TL;DR이 본문보다 먼저 결정되어야 한다
질문을 만들 때 **TL;DR을 한 문장으로 먼저 쓰고**, 그 다음에 그 답이 왜 그러한지를 풀어쓰는 본문 블록을 만든다. TL;DR이 흐릿하면 그 질문은 폐기.

### 2. 한 질문은 한 가지만 답한다
"X의 의미와 Y의 차이점, 그리고 Z의 한계"를 하나의 질문에 묶지 말 것. 쪼개라.

### 3. 블록 수 제한
질문당 본문 블록 2~5개. 6개 이상이면 그 질문은 너무 큰 주제다 — 두 개로 쪼개거나 dissection 카드로 보낸다.

### 4. 카테고리 A는 도메인 진입점
독자가 이 분야를 처음 본다고 가정. ViT, Token, KV-Cache, LoRA 같은 기초 개념을 묻는 데 부담이 없어야 한다. **단, 논문에서 직접 사용하는 개념만**.

### 5. 카테고리 B는 인과 사슬
"왜 이 순서인가" / "왜 이 단계만 deep층에서 하는가" / "왜 이 트릭이 정합성을 보장하는가" — 모두 dissection의 Execution Logic / Hidden Assumptions과 연결되어야 한다.

### 6. 카테고리 C는 표/그림 읽기
"Table 1에서 X가 유독 떨어지는 이유" / "Figure 9의 speedup이 모델 크기에 따라 다른 이유" — 본문이 명시한 결과를 **다시 읽어주는** 게 아니라, **암시만 한 함의를 짚어주는** 질문이어야 한다.

### 7. 커스텀 위젯은 한두 개만
Q1에 패치 → 1D 시퀀스 변환 애니메이션 같은 인터랙티브 위젯은 카테고리 A의 첫 질문 정도에 한 번 정도 둔다. 모든 질문에 위젯을 달면 본질이 흐려진다.

**위젯 클래스 규약 (`rules/component_rules.md` 참조):**
- 논문 prefix를 붙임 (예: `q1-patch`, `q1-token`, `q3-bar`)
- vanilla JS, 외부 라이브러리 의존 없음
- 위젯 마크업은 `html` 블록의 `content` 안에 인라인

## 분량 가이드

- 카테고리 3개 (A/B/C 고정)
- 카테고리당 질문 2~3개
- 전체 6~10 질문
- 질문당 블록 2~5개
- TL;DR 1~2문장
- 본문 블록 1개당 평균 2~5문장 또는 5×5 이하 표

전체 `qa.json`은 30~80KB 정도가 적정 (그 이상이면 너무 무겁다).

---

## 절대 금지

- 평가 시험지 모드 ("정답을 맞춰라")
- 논문 본문을 그대로 옮긴 답
- 카테고리 A에 갑자기 ablation 표 들이대기 (그건 C로)
- 위젯 7개 / 슬라이더 5개 같은 화려함
- TL;DR 없이 블록부터 시작
- block type 4종 외 추가 (`video`, `iframe`, `quiz` 등 금지)
- callout `variant`를 임의로 추가 (3종 고정)

## 이미지 생성

> Claude Code의 Bash tool로 `codex ...` 명령을 직접 실행해 ImageGen으로 PNG를 만든다. 별도 플러그인·MCP 자동화 없이 Bash 한 줄. 결과는 곧장 base64로 박아 `<figure class="concept-figure">` 정본 컴포넌트로 인라인 — 검수는 사후.
>
> **정식 호출 형식 (5계명 — Bash·UTF-8 prompt.txt·ASCII 인자·stdin null·스타일/출력 명시) + 검증된 명령 템플릿: `rules/component_rules.md` §11.**

이 단계에서 생성 가능한 이미지:
- 카테고리 A의 **도입 일러스트** (도메인 큰 그림 — 예: "비디오 → 토큰 변환의 길")
- 각 질문의 **TL;DR 보조 이미지** (한 장으로 답을 시각화)
- 카테고리 C의 **표·그림 해석 보조** (예: "Figure 9가 무엇을 말하는가"를 단순화한 일러스트)

저장 / 임베드:
- 저장 경로: `papers/[name]/assets/generated/`
- 파일명 규약: `qa_<qid>_<purpose>.png` (예: `qa_q1_intro.png`, `qa_q7_table_reading.png`)
- HTML 임베드: `qa.json#blocks`의 `html` 블록 안에 `<img>` — **Stage 10에서 base64 인라인으로 변환** (CLAUDE.md 자산 임베딩 정책). `<img src="assets/generated/...">` 외부 참조는 개발 미리보기 한정.
- **인터랙티브 애니메이션 위젯은 JS로 구현** (예: `q1-patch`, `q1-token` stepper). ImageGen은 정적 보조 일러스트에만.

질문당 이미지 0~1개가 적정. 모든 질문에 이미지를 붙이면 학습 흐름이 끊긴다.
